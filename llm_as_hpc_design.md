# LLM-as-超算：协议与原型

## 核心设计

### 角色

| 组件 | 谁负责 | 做什么 |
|------|--------|--------|
| **数值引擎** | Python | 跑适应性计算、种群演化、输出轨迹 |
| **LLM顾问** | DeepSeek (当前模型) | 分析轨迹、推荐策略调整、猜测隐藏参数关联 |
| **用户** | 你 | 裁定方向、驳回荒谬建议、决定什么时候跑 |

### 协议：三步迭代

```
[步1] 数值引擎跑N代 → 输出轨迹摘要
[步2] → 封装prompt → LLM分析 → 推荐N个演化方向
[步3] → 数值引擎按推荐参数跑 → 回到步1
```

### Prompt模板 (步2用)

```

- 方向: {direction_name}

- 当前best={best}, avg={avg}, std={std}

{best_params_diff}

{COUPLING_RULES_STR}
```

## 预期瓶颈

1. **成本** — 每50代调用一次LLM, 1000代约20次调用, 可以接受
2. **延迟** — LLM推理每次≈3-5s, 与数值计算并行
3. **幻觉** — LLM可能推荐生物学上不存在的参数组合 → 数值引擎自动验证fitness, 差的自然淘汰
4. **上下文窗口** — 30参数轨迹约500token/次, 够用

## 跟真超算对比

| 维度 | 真超算 | LLM超算 |
|------|--------|---------|
| 每次迭代 | 50代 × 50个体 | 1次推理(几秒) |
| 搜索方向 | 随机突变 | 定向推荐 |
| 新发现 | 意外的极端值 | 反直觉策略 + 生物知识 |
| 瓶颈 | CPU/内存 | 模型知识边界 |
| 最适场景 | 大规模枚举 | 策略指导 + 模式发现 |
